网关

一句话本质
网关是本地网络的出口守卫，替不认识路的数据包开门，顺手把地址换成外界认识的样子

第一幕：问题感知

你家里有五台设备：手机、笔记本、平板、智能电视、台式机。它们通过 Wi-Fi 连在一起，互相传文件没问题。

现在你想用手机刷 B 站。

B 站的服务器在北京某个机房，IP 地址是 119.3.70.188。你的手机 IP 是 192.168.1.5。

手机把请求数据包发出去——然后呢？数据包飞向哪里？

你的路由器？对。但路由器凭什么知道这个包该往哪送？它认识 119.3.70.188 吗？

你家里的网络只有五台设备。路由器的"地图"里，只有这五台设备的地址。119.3.70.188 不在这张地图上。路由器查了一圈，没找到——

数据包该扔掉吗？

生活中有没有这样一种情况：你不知道怎么去某个地方，但你知道去问谁？

不认识目的地，数据包怎么出门？

想一想：你会怎么设计这个"出门"机制？有没有一个最简单粗暴的方案？

第二幕：引导式构建

最直觉的解法——指定一个"出口"

你大概已经想到了：既然路由器不认识目的地，就给它定一条规则——"遇到不认识的地址，一律送去这里"。

这个"这里"，就是默认网关（Default Gateway）。

网关这个词，字面意思就是"网络的门"。当你的设备想把数据包送往一个"本地网络之外"的地址，它不会傻傻地广播问"谁是 119.3.70.188"，而是直接把包扔给网关——让网关来操心后面的事。

你家里的网关，通常就是你的路由器，IP 一般是 192.168.1.1。

用一个日常类比：你住在小区里，想寄一封信到外地。你不需要知道收件人在哪条街、哪栋楼。你只需要找到小区门口的邮筒（网关），把信扔进去。邮局的人会接着处理。

想一想：你的手机怎么知道网关的 IP 是多少？这个信息是怎么来的？

答案是：自动告诉你的。你的手机连上 Wi-Fi 的瞬间，路由器通过一个叫 DHCP 的机制，把三样东西一起发给你：你的 IP 地址、子网掩码、还有网关地址。你的手机压根不需要手动填。

好，数据包到了网关，然后呢？

网关（路由器）收到了你手机发来的包，目标是 119.3.70.188。它认识这个地址吗？

不一定认识。但它比你的手机更接近"外面的世界"——它连着运营商的网络。如果它也不认识，它会把包扔给它自己的上游网关，就是运营商的路由器。

这是一个层层传递的过程。每一跳，数据包都被送给"比我更靠近目标"的人，直到最终到达目的地。

这个逐跳转发的机制叫路由（Routing）。网关是路由链条的第一跳。

想一想：如果每一跳都"不认识目标就往上扔"，这条链条会不会永远传递下去、没有终点？什么时候才能到达目的地？

不会无限传递。互联网的骨干路由器维护着一张巨大的路由表，知道"去某段 IP 地址，该往哪个方向走"。越靠近核心，路由器知道的越多。最终总会有一台路由器认识目标地址，把包直接送达。

网关只是"出口"吗？还是做了更多事？

到这里，网关听起来只是个"转发站"——收到包，扔给上游。但家用路由器作为网关，其实还悄悄做了一件大事。

你家五台设备的 IP 都是 192.168.x.x——这类地址叫私有地址（Private Address），全世界有无数个家庭网络都在用完全相同的 192.168.1.x 段。这些地址在互联网上毫无意义，全球的路由器不认它。

但你的数据包需要出门，包头上的源地址是 192.168.1.5——服务器收到后怎么把回包发给你？它根本不知道 192.168.1.5 在哪。

网关在数据包出门的瞬间，把源地址从 192.168.1.5 悄悄改成了自己的公网 IP（比如 101.88.32.7），同时记下"这个改动是替哪台设备做的"。回包到来时，再把目标地址改回 192.168.1.5，送还给你的手机。

这个操作叫NAT（Network Address Translation，网络地址转换）。

想一想：路由器要"记下替哪台设备做的"——它记的是什么？只记 IP 地址够用吗？

只记 IP 不够——四元组才能唯一标识一条连接。路由器记的是完整的映射：(内网IP:内网端口) <-> 公网端口。回包来了，看目标端口，查表，找到对应的内网设备，改地址，转发。

内网里的数据包，走网关吗？

有一个很自然的问题：你的手机给同一个 Wi-Fi 下的笔记本发文件，也要经过网关吗？

不用。这正是网关只管"出门流量"的关键。

你的设备在发出数据包之前，先做一个判断：目标地址是不是在我的本地网络里？判断的工具就是子网掩码（Subnet Mask）。

用最直觉的方式理解：子网掩码划定了"我家的范围"。IP 地址 192.168.1.5，子网掩码 255.255.255.0，意味着 192.168.1.0 到 192.168.1.255 都是"自己人"——直接在本地网络内投递，不走网关。

目标不在这个范围？走网关，出门。

想一想：如果子网掩码配错了，把范围设得特别小，会发生什么？比如错误地认为连路由器都不是"自己人"——数据包还能出去吗？

有趣的问题。如果设备认为网关 IP 也不在本地网络，它会尝试把发给网关的包再次转发……给网关。但网关 IP 本来应该是直接可达的，这种错误配置会导致连路由器都连不上，彻底断网。子网掩码配错是真实存在的经典故障场景。

端到端示例：手机请求 B 站

场景设定

手机 IP:       192.168.1.5
子网掩码:       255.255.255.0  (本地范围: 192.168.1.0 - 192.168.1.255)
默认网关:       192.168.1.1
路由器公网IP:   101.88.32.7
B站服务器:      119.3.70.188:443

第一步：判断目标是否在本地

手机检查：119.3.70.188 在 192.168.1.x 范围内吗？不在。走网关。

第二步：数据包发给网关

手机构造数据包，目标 MAC 地址填路由器，送出去：

源IP:  192.168.1.5 : 54321
目标IP: 119.3.70.188 : 443

第三步：网关做 NAT，记录映射

路由器把包改写，并记录：

改写后:
源IP:  101.88.32.7 : 60001   <-- 公网端口由路由器分配
目标IP: 119.3.70.188 : 443

映射表新增一条:
60001 <-> 192.168.1.5:54321

第四步：逐跳路由到 B 站

包在互联网上跳若干次，到达 119.3.70.188。B 站服务器看到的源地址是 101.88.32.7:60001，回包原路打回。

第五步：网关反向 NAT，送回手机

路由器收到回包，目标是 101.88.32.7:60001，查映射表：找到 192.168.1.5:54321，改写目标地址，转发给手机。

第六步：手机收到数据

手机看到目标端口 54321，交给浏览器，B 站页面渲染出来。

第四幕：验证与压力测试

反事实：如果没有默认网关会怎样？

设备只能和本地网络内的设备通信。发给任何"外部地址"的包，都会因为"不知道送给谁"而被丢弃。你的电脑变成了一座孤岛——能在局域网里传文件，但上不了互联网。

这不是假设场景。配置服务器时如果忘填默认网关，机器能被同机房其他机器 ping 通，但访问不了外网，这是常见的运维失误。

边界：网关在哪些情况下不够用、或会成为瓶颈？

NAT 的单向性问题：NAT 只对"由内向外发起"的连接有效。外部设备主动连进来找你，路由器不知道该把包送给哪台内网设备——这就是为什么家里的电脑默认不能被外网直接访问。要绕过这个限制，需要端口转发（Port Forwarding）或者 NAT 穿透。

单点故障：一个局域网只有一个默认网关，网关挂掉，整个局域网断网。企业网络会配置双网关做冗余，家用网络通常没有。

多出口的困境：如果一个网络有两条出口线路（比如联通+电信），默认网关只能指向一个。去联通的流量走联通，去电信的流量也被迫走联通，绕了一大圈。解决这个问题需要策略路由（Policy Routing），按目标地址或流量类型选择不同出口。

连接：网关跟哪些概念相关？

  • 端口：NAT 映射表里记的就是完整四元组，端口是 NAT 工作的基础。没有端口，NAT 无法区分同一台内网设备发出的不同连接
  • DNS：你访问 B 站，输入的是 bilibili.com，不是 IP。DNS 把域名解析成 IP——这个 DNS 查询本身也要经过网关出去。没有网关，DNS 也用不了
  • 防火墙：企业网关通常同时兼任防火墙，在 NAT 的基础上加上规则过滤。"网关"在企业场景下是一个功能集合，不只是转发
  • VPN：VPN 的本质是在你和 VPN 服务器之间建一条加密隧道，然后把默认网关改成 VPN 的隧道出口——所有流量先走隧道，再从 VPN 服务器出去。理解网关，就理解了 VPN 改变流量路径的核心机制

终极挑战·空白重建

如果忘掉一切，只记得第一幕的问题，能重新发明网关吗？

[问题]
  数据包要去本地网络以外的地址
  设备不认识目标, 不知道该把包送给谁

[缺口1]
  需要一个"兜底规则": 不认识就送去哪里

[工具1: 默认网关]
  指定一个出口设备
  所有"外部流量"一律先送给它

[缺口2]
  设备怎么判断目标是"内部"还是"外部"?

[工具2: 子网掩码]
  划定本地网络范围
  在范围内 -> 直接投递
  不在范围 -> 走网关

[缺口3]
  内网私有地址在互联网上无效
  服务器无法把回包送回来

[工具3: NAT]
  网关出门时替换源地址为公网IP
  记录映射表
  回包时反向翻译, 送回内网设备

[完整方案]
  子网掩码判断内外
  -> 外部流量走默认网关
  -> 网关 NAT 改写地址出门
  -> 逐跳路由到目标
  -> 回包经 NAT 反查送回

只凭这张地图，你能独自走完全程。网关就是这么一件事：本地网络的出口守卫，替不认识路的数据包开门，顺手把地址换成外界认识的样子。
