Node.js

一句话本质
Node.js 用单线程事件循环 + 非阻塞 I/O 实现高并发：把等待任务交给操作系统，自己继续转，用 Promise/async/await 优雅表达异步流程。

第一幕：问题感知

你写了一个网页，里面有个按钮，点击之后会发一条消息给服务器，服务器把消息存起来，然后告诉你"存好了"。

很简单的事情。你开始写服务器端的代码。

服务器跑起来了。第一个用户连进来，发了一条消息，服务器处理，回复，没问题。

第二个用户同时连进来了。

你的服务器在处理第一个用户的时候，需要去数据库里写一条记录。写数据库这个动作，要花几十毫秒。在这几十毫秒里，服务器在等——等数据库写完。

第二个用户的请求，就在门口站着，进不来。

你说：那我多开几个"工人"，每个用户来了，就派一个工人专门服务他。

可以。但每个工人都要占内存，都要消耗系统资源。如果同时来了一万个用户，就要养一万个工人——光是这些工人站在那里喘气，内存就撑不住了。

而且，大多数时候，这一万个工人其实都在等——等数据库、等文件读写、等网络响应。真正在干活的时刻少之又少。一万个工人，九千九百个在发呆。

一个工人，怎么同时服务一万个请求？

第二幕：引导式构建

想象一个咖啡店的收银员。

顾客 A 点了一杯需要等五分钟的手冲咖啡。收银员不会站在那里等五分钟——他接过订单，把它交给咖啡师，然后立刻转头去处理顾客 B。等咖啡师喊"好了！"，他再回来把咖啡端给顾客 A。

这个收银员从来没有傻等。他一直在动，但他只有一个人。

这就是 Node.js 的核心思路。

Node.js 里有一个东西叫 事件循环（Event Loop）——它是 Node.js 唯一的"工人"，但它从不傻等。它的工作方式是：把所有"需要等待的事"（读文件、查数据库、发网络请求）交给操作系统去做，自己继续处理下一件事。

想一想：操作系统做完了，怎么通知事件循环"我好了，轮到你了"？

这个通知的载体，叫做 回调函数（Callback）——你在交出任务的时候，同时附上一张"便条"：等你做完，执行这段代码。操作系统做完之后，把这张便条放进待处理队列，事件循环空闲了就去取。

你的代码                事件循环               操作系统
    |                      |                      |
    | 读文件，附上便条 A     |                      |
    |--------------------->|                      |
    |                      | 把读文件交给 OS        |
    |                      |--------------------->|
    |                      |                      | 正在读...
    | 处理请求 B             |                      |
    |--------------------->|                      |
    |                      | 处理请求 B            |
    |                      |                      | 读完了！
    |                      |<---------------------|
    |                      | 取出便条 A，执行       |

这个模型有个专门的名字：非阻塞 I/O（Non-blocking I/O）——"阻塞"就是傻等，"非阻塞"就是把任务交出去，继续干别的，等通知。

想一想：如果某个任务不是"等待 I/O"，而是需要大量计算的代码（比如对一百万个数字排序），会发生什么？

这是事件循环的软肋。计算密集的任务没法"交出去等"——事件循环必须自己一行一行算，在这段时间里它被卡住了，所有其他请求只能排队。

这就是 Node.js 最重要的设计边界：它非常擅长 I/O 密集型的任务（读写文件、网络请求、数据库），但对 CPU 密集型的任务（图像处理、复杂计算）不擅长。

想一想：非阻塞 I/O 解决了"等待"的问题，但如果多个异步操作之间有先后依赖——比如先查用户，再用查到的 ID 去查订单——你会怎么写这段代码？

最直接的写法，是在回调里再发起下一个请求，拿到结果再嵌套下一个：

查用户(function(用户) {
    查订单(用户.id, function(订单) {
        查商品(订单.商品id, function(商品) {
            // 终于到了
        });
    });
});

代码越来越往右缩进，像一个直角三角形——这个形状，JavaScript 社区叫它 回调地狱（Callback Hell）。读起来痛苦，出了问题更难调试。

想一想：嵌套的根本问题在哪儿？"先做 A，再做 B，再做 C"这个先后顺序，一定要用嵌套来表达吗？

不一定。嵌套只是一种表达方式——把"先后顺序"和"嵌套层级"绑定在一起，是回调写法的局限，不是异步本身的限制。

如果有一种方式，能把"先后顺序"表达成一条直线而不是嵌套，问题就解决了。

Promise 就是这个方案。

Promise 是对"将来会有结果"这件事的一个包装对象——你不用马上得到结果，你先拿到一个"承诺"，然后用 .then() 告诉它"等你有结果了，做这件事"，.then() 本身也返回一个新的 Promise，于是可以继续串：

查用户()
    .then(用户 => 查订单(用户.id))
    .then(订单 => 查商品(订单.商品id))
    .then(商品 => {
        // 同样的逻辑，直上直下
    });

嵌套消失了，变成了一条链。逻辑顺序和代码顺序对齐了。

想一想：这条链上如果某一步出错了，需要每一步都写错误处理吗？

不用。在链的末尾加 .catch()——链上任何一步抛出的错误，都会跳过后续 .then()，直接落到 .catch()。一个兜底，管整条链。

Promise 已经好很多了。但很多人觉得 .then().then().then() 读起来仍然不像正常人说话的方式。于是出现了 async/await。

"语法糖"（Syntactic Sugar）是编程里的一个说法——用更甜、更好读的语法包裹了底层机制，让代码看起来更自然，但背后什么都没变。async/await 就是 Promise 的语法糖：背后还是 Promise，但写起来像同步代码：

async function 处理请求() {
    const 用户 = await 查用户();
    const 订单 = await 查订单(用户.id);
    const 商品 = await 查商品(订单.商品id);
    // 直上直下，和同步代码一样读
}

await 的意思是：等这个 Promise 完成，把结果拿出来给我。async 标记这个函数里会用 await。

想一想：await 会让事件循环停下来等吗？

这是 async/await 最关键的一点：不会。

当执行到 await 时，Node.js 把这个函数当前的执行位置"存档"（就像游戏存档一样），然后把控制权还给事件循环——让它继续处理其他请求。等 Promise 完成，Node.js 取出"存档"，从 await 的下一行继续执行。

整个过程中，事件循环从未停止转动。await 只是让这个函数暂停，而不是让整个程序暂停。这是它看起来像同步、实际上是异步的秘密。

第三幕：完整组装

+--------------------------------------------------------+
|                     你写的代码                           |
|  async/await  -->  Promise  -->  Callback              |
|  (语法糖，最易读)    (包装对象)    (原始通知机制)           |
+--------------------------------------------------------+
                          |
                    交出 I/O 任务
                          |
                          v
+--------------------------------------------------------+
|                    事件循环 (Event Loop)                 |
|  "我不等，我继续转。有结果了通知我。"                        |
+--------------------------------------------------------+
          |                              ^
          | 交出任务                       | 完成通知（回调入队）
          v                              |
+--------------------------------------------------------+
|              操作系统 / 底层 I/O (libuv)                 |
|   文件读写 / 网络请求 / 数据库 / DNS ...                   |
+--------------------------------------------------------+

端到端示例：一个最小的 HTTP 服务器

// server.js
const http = require('http');
const fs = require('fs/promises');  // Promise 版本的文件操作

const server = http.createServer(async (req, res) => {

    // await 暂停这个函数，把控制权还给事件循环
    // 事件循环继续处理其他请求
    const content = await fs.readFile('./data.txt', 'utf-8');

    // 文件读完了，从这里继续
    res.writeHead(200, { 'Content-Type': 'text/plain' });
    res.end(content);
});

server.listen(3000, () => {
    console.log('服务器跑起来了');
});

同时来了 3 个请求时，发生了什么：

时间轴:
t=0ms    请求A进来  -> await 读文件A -> 事件循环继续
t=1ms    请求B进来  -> await 读文件B -> 事件循环继续
t=2ms    请求C进来  -> await 读文件C -> 事件循环继续
t=20ms   文件A读完  -> 事件循环回来  -> 回复请求A
t=21ms   文件B读完  -> 事件循环回来  -> 回复请求B
t=22ms   文件C读完  -> 事件循环回来  -> 回复请求C

一个事件循环，交错处理三个请求。没有任何一个请求需要等另一个处理完。

各组件对应位置：

概念 在示例中的位置
事件循环 Node.js 运行时内部，隐式运转
非阻塞 I/O fs.readFile 交给操作系统，不卡住
async/await async (req, res) + await fs.readFile
Promise fs/promises 返回的就是 Promise
回调 server.listen(3000, () => {...}) 里的箭头函数

第四幕：验证与压力测试

反事实：如果 Node.js 用传统的多工人模型

  • 1 万个并发连接 = 1 万个工人
  • 每个工人约占 1-8MB 内存，1 万个工人 = 10GB+ 内存
  • 大量工人在 I/O 等待期间什么都不做，白占资源
  • Node.js 的单线程事件循环在 I/O 密集场景下，用几十 MB 内存处理同样的并发量

反事实：如果没有 Promise，只有回调

  • 嵌套地狱：5 个有依赖关系的异步操作 = 5 层缩进
  • 错误处理：每一层都要单独 if (error) ...，漏掉任何一层都是 bug
  • 代码流程难以追踪：逻辑被分散在多层回调里

边界：Node.js 在哪些情况下不适合

CPU 密集型任务会卡死事件循环。 视频转码、大规模数据计算——这类任务一旦开始，事件循环被占用，所有其他请求全部阻塞。

解决方案是 Worker Threads（工作线程）——Node.js 允许创建真正的额外线程处理计算密集型任务，把它们从事件循环里移出去。

单点故障风险。 单线程意味着如果代码有未捕获的异常，整个进程可能崩溃。生产环境通常用 PM2 这类进程管理工具来监控和自动重启。

连接：Node.js 与已学知识的关系

Node.js 和 npm/pnpm 的关系：npm 是 Node.js 的包管理器，随 Node.js 一起安装。package.json、node_modules、npm install——这些都是 Node.js 生态的基础设施。没有 Node.js，npm 没有意义。pnpm 也运行在 Node.js 之上。

Node.js 和 SDK 的关系：几乎所有云厂商的 SDK（腾讯云、AWS……）都提供 Node.js 版本。SDK 里的 Client / Signer / Retry——在 Node.js 版本里全部用事件循环 + async/await 实现异步调用。
