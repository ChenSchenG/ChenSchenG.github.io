npm（Node Package Manager）

一句话本质
npm 是 JavaScript 的包管理器：用 package.json 声明依赖意图，从公共 registry 自动下载到 node_modules，用 package-lock.json 锁定精确版本。

第一幕：问题感知

你刚入职一家公司，拿到一台新电脑。

同事发给你一个项目的代码压缩包，你解压，打开文件夹——代码文件加起来不到 200KB，很轻。你双击 index.js，什么也没发生。你问同事："怎么跑起来？"

同事说："你得先把依赖装上。"

你愣了一下。"依赖？"

同事解释：这个项目用到了别人写好的工具库，比如处理日期的、格式化颜色的、解析命令行的——作者没有把这些工具的代码一起打包给你，因为那些工具库本身又依赖了另一些工具库，那些库又依赖了更多库。如果全部手动复制粘贴，光是理清"谁依赖谁、哪个版本"这件事，就能让人崩溃。

同事打开终端，输入一行命令。几秒后，项目跑起来了。

你盯着那行命令，忽然意识到：

如何让"把所有依赖按正确版本装好"这件事，变成一行命令？

第二幕：引导式构建

假设你的项目只需要一个外部工具：chalk（一个给终端输出加颜色的库）。

手动做法：去 GitHub 找到 chalk，下载代码，放进你的项目文件夹，手动处理它的依赖……

等等，chalk 本身也有依赖。它的依赖的依赖呢？

在你绝望之前，先退一步，只解决"记录我用了什么"这个小问题。

你创建一个文件，叫它 package.json。它只做一件事：用结构化的方式写清楚"这个项目叫什么、需要哪些外部工具、每个工具要哪个版本"。

{
  "name": "my-project",
  "dependencies": {
    "chalk": "5.0.0"
  }
}

想一想：有了这份清单，"帮我把清单上的东西都装好"这件事，是不是就可以交给一个自动化工具去做了？

工具库不在你的电脑上，也不在同事的电脑上。它们需要一个公共的存放地点——一个所有人都能上传、也能下载的仓库。这个仓库就叫 registry（注册表），可以把它想象成一个公共图书馆：任何人都可以把自己写的工具"捐"进去，任何人也都可以来借。

想一想：下载下来放哪儿？总不能和你的源代码混在一起——不然别人拿到你的代码，又要重新厘清哪些是你写的、哪些是下载的。

你给外部工具指定一个统一的落脚点：node_modules 文件夹。所有从公共图书馆下载的工具，都放进这里。你的同事拿到项目时不需要这个文件夹——他只要拿到 package.json，让自动化工具重新下载一遍就好。node_modules 是可以随时删掉、随时重建的。

现在还有一个隐患。你在 package.json 里写 "chalk": "^5.0.0"，意思是"5.x.x 里任何新版本都行"。今天你装的是 5.0.1，三个月后同事重新装，可能装到了 5.3.0。代码行为悄悄变了，却没人发现。

于是出现了 package-lock.json——它记录的不是"我想要什么版本"，而是"我上次实际装了什么版本，精确到每一个间接依赖"。这是一份快照，保证任何人在任何时候执行安装，都能得到和你完全一样的结果。

想一想：package.json 是意图，package-lock.json 是现实。两份文件都应该提交到代码仓库吗？

都应该提交。package.json 告诉别人"这个项目要什么"，package-lock.json 保证"装出来的结果和你一模一样"。只提交前者，版本可能飘；只提交后者，别人看不懂意图。

第三幕：完整组装

你的项目
|
+-- package.json          <- 意图清单（"我需要这些工具"）
+-- package-lock.json     <- 精确快照（"上次实际装了这些版本"）
+-- index.js              <- 你写的代码
|
+-- node_modules/         <- 所有外部工具的落脚点（不提交到 git）
    +-- chalk/
    +-- ansi-styles/      <- chalk 的依赖，也被装进来了


                    +-----------------------------+
                    |       npm registry          |
                    |   (公共图书馆，在云端)        |
                    +-------------+---------------+
                                  | npm install
                                  v
+--------------------------------------------------+
|  你的电脑                                          |
|  package.json  -->  告诉 npm 要装什么              |
|  package-lock.json <-- npm 写入精确版本快照         |
|  node_modules/ <--- npm 把文件下载到这里            |
+--------------------------------------------------+

端到端示例：从零开始创建一个项目

# 初始化，生成 package.json
npm init -y

# 安装一个工具库
npm install chalk
# 1. 去 registry 下载 chalk 及其所有依赖
# 2. 放入 node_modules/
# 3. 更新 package.json（dependencies 里加了 chalk）
# 4. 生成/更新 package-lock.json

# 在代码里使用
# index.js
import chalk from 'chalk';
console.log(chalk.blue('Hello!'));

node index.js   # 输出：Hello!（蓝色）

拿到别人的项目，恢复所有依赖

npm install
# npm 读取 package-lock.json，精确还原，版本和原作者完全一致

关键文件对照表

文件 作用 提交到 git？
package.json 声明项目信息与依赖意图 ✅ 必须
package-lock.json 锁定实际安装的精确版本 ✅ 必须
node_modules/ 实际下载的工具文件 ❌ 不要

常用命令

命令 做了什么
npm init -y 生成默认 package.json
npm install <包名> 安装一个包，加入 dependencies
npm install 按 lock 文件还原所有依赖
npm uninstall <包名> 卸载一个包
npm run <脚本名> 运行 package.json 里定义的脚本

第四幕：验证与压力测试

反事实：如果没有 npm

你需要手动去每个工具库的 GitHub 页面下载代码、手动追踪"A 依赖 B 的 1.2 版，B 依赖 C 的 2.0 版"这张依赖图、每次换电脑或换同事都要重新做一遍。npm 把这些全部自动化了。

边界：npm 不能做什么

npm 不保证代码是安全的。 任何人都可以往 registry 发布包，历史上出现过恶意包混入的情况（名字和知名库只差一个字母）。npm audit 可以扫描已知漏洞，但不是万能防火墙。

npm 不解决"同一个包被装了十份"的问题。 如果你的电脑上有 50 个项目，每个都用了 lodash——硬盘上存了 50 份完全相同的文件。这个问题由 pnpm 解决。
