OIDC（OpenID Connect）

一句话本质
OIDC 是 OAuth 2.0 的超集：在委托授权之上加一层标准化身份层，用 JWT 格式的 ID Token 让任意客户端无需额外请求即可知道"你是谁"。

第一幕：问题感知

你用 GitHub 账号登录了一个第三方网站，比如 Notion。

点击"用 GitHub 登录"，跳转到 GitHub，你同意授权，跳回来——你就登进去了。

背后用的是 OAuth 2.0。GitHub 给 Notion 发了一个 Access Token。

现在，Notion 拿着这个 Access Token，能干什么？

它能以你的名义访问 GitHub API——比如读你的仓库列表、读你的邮箱。这是 OAuth 2.0 设计的本意：委托操作权限。

但 Notion 真正想要的，其实不是操作你的 GitHub 数据。

Notion 想知道的只有一件事：你是谁？

具体来说：你的用户名是什么、你的邮箱是什么、你的头像是什么——这些信息足够 Notion 为你建立账号、让你登录进去。

但 OAuth 2.0 只管"你有没有权限做某事"，它不管"你是谁"。Access Token 是一把钥匙，能开门，但门上没写主人的名字。

Notion 拿到 Access Token 之后，如果想知道你是谁，它还得额外调一次 GitHub 的 /user API——用 Token 去问 GitHub："这个 Token 对应的用户叫什么？"

这个方案能用，但有个问题：每家平台的"身份接口"长得不一样。GitHub 叫 /user，Google 叫 /userinfo，微博又是另一个 URL，返回的字段名也各不相同。

如果你想同时接入 GitHub、Google、微博登录，你要写三套对接代码。

有没有一套统一的、标准的方式，让"用 X 账号登录"这件事不依赖于每家平台的私有接口？

如何在委托授权的同时，顺手知道"你是谁"？

第二幕：引导式构建

OAuth 2.0 解决了"委托授权"的问题，但它刻意没有定义身份层。

规范里明确说：Access Token 是不透明的（opaque）——授权服务器发什么格式都行，接收方不应该自己解读 Token 里面有什么。

所以在纯 OAuth 2.0 的世界里，Notion 拿到 Access Token 之后，并不知道这个 Token 对应的是哪个用户。它必须额外去问。

这"额外去问"就是问题所在：没有标准，每家不一样，对接成本高。

OIDC 要做的事情，就是在 OAuth 2.0 的框架上，加一层标准化的身份层。

想一想：如果你来设计这个身份层，你会怎么做？OAuth 2.0 已经有一个 Access Token 了，你会直接扩展它，还是另外再发一个东西？

OIDC 的答案是：再发一个专门代表身份的 Token。

这个新 Token 叫做 ID Token（身份令牌）。ID Token 和 Access Token 并排存在，各司其职——Access Token 代表你有什么操作权限，ID Token 代表你是谁。职责分离，不相互侵入。

那 ID Token 长什么样？

这里 OIDC 做了一个关键选择：ID Token 必须是 JWT 格式。

你已经学过 JWT——它是 Header.Payload.Signature 三段结构，Payload 里放 JSON 字段，整体用签名防篡改。

选 JWT 的好处是：Notion 拿到 ID Token 之后，不需要再去问 GitHub，自己就能解开读取身份信息，同时验证签名确认这个 Token 确实是 GitHub 签发的。

想一想：这里有个重要的前提——Notion 要验证 ID Token 的签名，它需要什么？

Notion 需要 GitHub 的公钥。签名是用私钥签的，验证要用对应的公钥。

OIDC 规定，每个身份提供方（比如 GitHub）必须在一个固定的、公开的 URL 上发布自己的公钥列表，这个 URL 叫做 JWKS 端点（JSON Web Key Set）。任何人都可以去这个 URL 拿到公钥，然后验证 ID Token 的签名是否合法。

现在 ID Token 能被验证了，接下来的问题是：Payload 里放哪些字段来代表身份？

OIDC 把 Payload 里的字段统称为 Claims（声明）——每一个字段，就是一条"关于这个用户的声明"。

OIDC 首先规定了一个最重要的字段：sub（Subject，主体）。sub 是用户在这个身份提供方的唯一 ID，不随时间变化。就像身份证号码——不管你改名换姓，这个编号不变。

有了 sub，Notion 就能知道"这是同一个人"。但光知道"是谁"还不够——Notion 还需要知道"这个 Token 是可信的吗？"

这就需要另外几个字段配合。iss（Issuer，签发者）记录这个 Token 是谁发的，比如 https://accounts.google.com——Notion 验证 iss 是不是自己信任的提供方。exp（Expiration，过期时间）和 iat（Issued At，签发时间）控制 Token 的有效期窗口——Notion 验证当前时间是否在 iat 到 exp 之间。

想一想：还缺一个什么字段？假设攻击者从网站 A 截获了你的 ID Token，拿去网站 B 登录——现有的 sub/iss/exp/iat 能阻止这件事吗？

阻止不了。Token 是真实的、没过期的、签发者也可信——但它不是发给网站 B 的。

所以还需要 aud（Audience，受众）字段：这个 Token 是发给哪个客户端用的（比如 Notion 的 Client ID）。网站 B 验证 Token 时，发现 aud 不是自己的 Client ID，直接拒绝。

这五个字段——sub、iss、aud、exp、iat——是所有 ID Token 都必须包含的核心 Claims。

除此之外，OIDC 还定义了一组可选的标准 Claims：name（姓名）、email（邮箱）、picture（头像 URL）、locale（语言偏好）……字段名是标准化的，所以不管你对接哪家 OIDC 兼容的提供方，同样的代码都能读取同样的字段。

想一想：知道了 ID Token 的字段结构，那 Notion 怎么告诉 GitHub"我需要 ID Token，而不只是普通的 Access Token"？

这里 OIDC 的设计非常巧妙：它复用了 OAuth 2.0 的 Scope（作用域）机制。

Scope 是客户端在授权请求里声明"我需要什么权限"的字段。OIDC 在这个体系里加了一个特殊的 Scope 关键字：openid。

服务器收到授权请求后，会检查 Scope 列表。如果看到了 openid，就知道"这不只是普通的授权请求，而是身份认证请求"——于是在响应里额外签发并返回 ID Token。如果没有 openid，就走纯 OAuth 2.0 流程，只返回 Access Token。

这个开关设计让 OIDC 向后完全兼容 OAuth 2.0：加一个词，升级成 OIDC；不加，就是普通 OAuth。

OIDC 还在 openid 之外定义了两个标准 Scope 来控制返回哪些字段：加上 profile，服务器就在 ID Token（或 UserInfo 端点）里附上 name、picture、locale 等个人资料；加上 email，就附上 email 和 email_verified。

想一想：一个请求里同时包含 openid、profile、email，还有一个自定义的 read:repos，会发生什么？

服务器看到 openid，知道要走 OIDC 流程，会签发 ID Token；看到 profile 和 email，知道 ID Token 里（或 UserInfo 端点）要包含对应的用户信息字段；看到 read:repos，知道要授权读取仓库的 Access Token。响应里会同时有 ID Token（身份）和 Access Token（权限），两件事一次完成。

现在还剩一个问题：ID Token 里放不了太多字段——JWT 要放在 URL 参数或 HTTP 头里传输，太大会有问题。但有时候客户端需要更详细的用户信息怎么办？

OIDC 定义了 UserInfo 端点（UserInfo Endpoint）：客户端拿着 Access Token 去请求这个端点，可以按需获取更详细的用户信息。返回的字段名和 ID Token 里的 Claims 完全一致，是同一套标准。

这样的设计思路是"轻量 Token + 按需拉取"——ID Token 只放验证身份必须的字段，需要更多信息再去 UserInfo 端点取。

最后一个问题：Notion 怎么知道 Google 的 JWKS 端点和 UserInfo 端点在哪？

OIDC 规定，每个身份提供方都必须在一个固定格式的 URL 上发布自己的配置文档：

{issuer}/.well-known/openid-configuration

比如 Google 的配置文档在 https://accounts.google.com/.well-known/openid-configuration。打开这个 URL，你会看到一个 JSON，里面列出了所有端点地址：授权端点、Token 端点、UserInfo 端点、JWKS 端点……

这个机制叫做 Discovery（发现）。客户端只需要知道 issuer 的 URL，就能自动发现所有端点，不需要硬编码。

想一想：Discovery 机制有什么安全隐患？

如果 Discovery 文档被篡改，攻击者可以把 JWKS 端点指向自己控制的地址，返回自己的公钥——这样攻击者自签的 ID Token 就能通过验证。所以 Discovery 文档本身必须通过 HTTPS 获取，并且要缓存合理的时间：不能每次请求都重新拉取（性能差），也不能缓存太久（无法及时感知密钥轮换）。

端到端示例：用 Google 账号登录 Notion

假设 Notion 的 Client ID 是 notion_client_123。

Step 1 — 构造授权 URL：

https://accounts.google.com/o/oauth2/v2/auth
  ?response_type=code
  &client_id=notion_client_123
  &redirect_uri=https://notion.so/callback
  &scope=openid%20profile%20email
  &state=random_csrf_string

注意 scope 里有 openid——这是触发 OIDC 的关键。

Step 2 — 用户登录 Google，同意授权。Google 重定向回：

https://notion.so/callback?code=AUTH_CODE_xyz&state=random_csrf_string

Step 3 — Notion 后端用 code 换 Token：

POST https://oauth2.googleapis.com/token
  code=AUTH_CODE_xyz
  client_id=notion_client_123
  client_secret=notion_secret_abc
  redirect_uri=https://notion.so/callback
  grant_type=authorization_code

Google 返回：

{
  "access_token": "ya29.xxx",
  "id_token": "eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiIxMTI5MDQ5MTU2MDUzMDc0MzM4MDQiLCJpc3MiOiJodHRwczovL2FjY291bnRzLmdvb2dsZS5jb20iLCJhdWQiOiJub3Rpb25fY2xpZW50XzEyMyIsImV4cCI6MTcxMTM5MjAwMCwiaWF0IjoxNzExMzg4NDAwLCJlbWFpbCI6InVzZXJAZ21haWwuY29tIiwibmFtZSI6IkFsaWNlIn0.SIGNATURE",
  "token_type": "Bearer",
  "expires_in": 3600
}

Step 4 — Notion 验证 ID Token：

先去 https://accounts.google.com/.well-known/openid-configuration 拿配置，找到 jwks_uri，再去取公钥。

然后：

  • 验证签名：用 Google 公钥验证 SIGNATURE 是否合法 ✓
  • 验证 iss：https://accounts.google.com ✓
  • 验证 aud：notion_client_123 ✓（是我们自己的 Client ID）
  • 验证 exp：没过期 ✓

全部通过。从 Payload 取出：

  • sub = "112904915605307433804"（Google 内部的用户唯一 ID）
  • email = "user@gmail.com"
  • name = "Alice"

Step 5 — Notion 用 iss + sub 组合作为用户唯一标识（防止不同提供方的 sub 碰撞），查找或创建账号，建立 Session，登录成功。

第四幕：验证与压力测试

反事实：去掉 ID Token，只用 Access Token 会怎样？

这正是纯 OAuth 2.0 的做法——Notion 必须额外调一次 /userinfo API。

问题有三：每次登录多一次网络请求；每家提供方的接口不一样，没有标准；Access Token 是不透明的，Notion 不应该自己解读它的格式。

ID Token 的价值在于：可本地验证、格式标准、自带身份信息——一次 Token 交换之后，不需要额外的网络请求就能完成身份认证。

反事实：ID Token 如果不用 JWT，改用随机字符串会怎样？

那 Notion 就无法本地验证了。它必须把 ID Token 发回给 Google 的端点问"这个 Token 是不是你发的，对应哪个用户"——回到了需要额外网络请求的老路。

JWT + 签名验证，是 OIDC 能做到"本地验证"的关键。

边界：OIDC 在哪些情况下会失效？

签名算法选错：OIDC 的 ID Token 通常用 RS256（非对称，RSA 签名）或 ES256（椭圆曲线）。有一种危险的配置：如果服务器同时接受 HS256（对称，HMAC），攻击者可以用 OIDC Provider 的公钥（本来只用于验证，是公开的）作为 HS256 的密钥来伪造 Token——因为 HS256 的"密钥"和"验证密钥"是同一个东西。这就是算法混淆攻击在 OIDC 中的具体应用。防御方法：验证方必须强制指定允许的算法列表，绝不接受 Token 里声明的算法作为信任来源。

sub 的稳定性假设失效：OIDC 约定 sub 是用户在该提供方的永久不变 ID。但现实中有些提供方的 sub 会变（账号合并、迁移）。如果 Notion 用 sub 关联账号，sub 变了就会出现"老用户被当新用户"的问题。

nonce 被忽略导致重放攻击：OIDC 支持在授权请求里带一个 nonce（随机值），这个值会原样嵌进 ID Token 的 Payload。客户端验证 ID Token 时，必须检查 nonce 是否和自己发出的一致。如果不检查，攻击者截获一个合法的 ID Token，可以重新提交给同一个客户端。

连接：OIDC 跟 SAML 是什么关系？

SAML（Security Assertion Markup Language）是 OIDC 的"前辈"，解决的是同样的问题：跨系统的身份联合（Federated Identity）。

维度 SAML OIDC
时代 2002 年，企业时代 2014 年，移动互联网时代
格式 XML JSON + JWT
适合场景 企业 SSO（内网系统互信） 消费级"社会化登录"、移动 App
复杂度 高（XML 签名和加密极其复杂） 中等（基于 OAuth 2.0，JSON 友好）

现实中，企业内部系统（Salesforce、Workday 等）大量使用 SAML；消费级"用 Google/GitHub 登录"基本都是 OIDC。
