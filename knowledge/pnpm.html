pnpm（Performant npm）

一句话本质
pnpm 用全局 store + 硬链接解决 npm 的重复存储问题，用严格目录结构消灭幽灵依赖——相同内容只存一份，只暴露你声明的包。

第一幕：问题感知

你已经用 npm 工作了半年，一切都挺顺。

某天，你的硬盘突然提示"存储空间不足"。你去查，发现有一个文件夹叫 node_modules，散落在你电脑上几十个项目文件夹里。光是这些 node_modules，加起来占了 18GB。

你打开其中三个项目的 node_modules，发现一件让你抓狂的事：

项目A/node_modules/lodash/  <- 532 KB
项目B/node_modules/lodash/  <- 532 KB  （一模一样）
项目C/node_modules/lodash/  <- 532 KB  （还是一模一样）

每一份都完全相同，每一份都独立占用硬盘。50 个项目，50 份 lodash。

同样的文件，为什么要在硬盘上存 50 次？

第二幕：引导式构建

在你想解决"同一份文件存了 50 次"之前，先看看操作系统是怎么处理类似问题的。

系统里不同的应用可能都需要同一套字体，但操作系统不会把字体复制成 100 份。它在一个地方存了字体，然后给每个需要它的应用一个指向那个位置的路径。

这种"我不复制文件，我只给你一个入口"的机制，叫做硬链接（hard link）。

硬链接不是快捷方式。快捷方式是"这里有一张纸条，告诉你真正的文件在哪儿"——删掉真正的文件，纸条就失效了。硬链接更像是"同一份文件，有两个门牌号"——通过任何一个门牌号访问，都到达同一块真实的数据；删掉一个门牌号，文件还在，另一个门牌号还能用。

想一想：如果 lodash 在硬盘上只存一份，50 个项目都通过硬链接指向它，那 18GB 里关于 lodash 的那部分，是不是可以缩减到几百 KB？

要让硬链接发挥作用，你需要一个"总仓库"——所有下载过的包，只在这里存一份真实文件。pnpm 把这个地方叫做 store，默认放在 ~/.pnpm-store。

每次安装一个包，pnpm 先检查 store 里有没有这个包的这个版本：有 → 不下载，直接从 store 建立硬链接；没有 → 下载到 store，再建立硬链接。

你的 50 个项目，node_modules/lodash 各自都有一个"入口"，但背后指向的都是 store 里那同一块数据。

想一想：硬盘占用降下来了。那速度呢？如果 store 里已经有了，根本不需要去网络下载——第二次安装同一个包，是不是瞬间完成？

是的。这是 store 机制的"意外收获"：速度快不只是因为网络快，而是根本不走网络。

pnpm 还顺手解决了 npm 的另一个问题：幽灵依赖（phantom dependency）。

用 npm 工作一段时间，你可能碰到过一个诡异情况：package.json 里明明没写 lodash，但代码里 import lodash from 'lodash' 却能跑。因为你装的某个工具库依赖了 lodash，npm 把 lodash 铺平放进了 node_modules/，Node.js 扫描时意外找到了它。

你没有声明这个依赖，但你用上了——这叫幽灵依赖。危险在于：某天那个工具库升级了，不再依赖 lodash，lodash 从 node_modules 消失，你的代码在毫无预警的情况下崩溃。

pnpm 用严格的目录结构消灭了这个问题：node_modules/ 的第一层只放你亲手在 package.json 里写的包。那些包的依赖被放在隐藏的 .pnpm/ 区域——包与包之间能互相找到彼此，但你的代码看不见没被声明的包。

想一想：严格结构意味着——如果你的项目依赖了幽灵依赖，切换到 pnpm 会立刻报错暴露。这是一种"把隐患提前引爆"的策略，你怎么看？

这正是 pnpm 的设计哲学：短期阵痛，换长期确定性。把隐患暴露在开发阶段，好过在生产环境某次升级后悄悄崩溃。

第三幕：完整组装

npm vs pnpm 的 node_modules 结构

npm 的 node_modules（铺平结构）：

node_modules/
+-- chalk/          <- 你声明的
+-- lodash/         <- chalk 的依赖，但你也能直接用（幽灵）
+-- ansi-styles/    <- 所有层级的依赖全部铺在第一层
+-- ...


pnpm 的 node_modules（严格结构）：

node_modules/
+-- chalk/          <- 符号链接 --> .pnpm/chalk@5.0.0/node_modules/chalk
|                                   （硬链接 --> ~/.pnpm-store 真实文件）
+-- .pnpm/          <- 内部区域，你的代码访问不到
    +-- chalk@5.0.0/
        +-- node_modules/
            +-- chalk/       <- 硬链接 --> store
            +-- ansi-styles/ <- 硬链接 --> store（chalk 的依赖，隔离在此）

全局 store 的作用

你的电脑
|
+-- ~/.pnpm-store/          <- 全局仓库（真实文件只存一份）
|   +-- lodash@4.17.21/
|   +-- chalk@5.0.0/
|
+-- 项目A/node_modules/chalk/ --+
+-- 项目B/node_modules/chalk/ --+--> 同一块数据（硬链接）
+-- 项目C/node_modules/chalk/ --+
              |
              v
    ~/.pnpm-store/chalk@5.0.0/

端到端示例

# 安装 pnpm（只需一次）
npm install -g pnpm

# 安装依赖
pnpm add chalk
# 1. 检查 store 里有没有 chalk 5.x
# 2. 没有 → 下载到 store；有 → 跳过下载（瞬间完成）
# 3. 在 .pnpm/ 建立严格结构
# 4. 在 node_modules/chalk/ 建立符号链接
# 5. 更新 package.json 和 pnpm-lock.yaml

# 幽灵依赖会被立刻暴露：
# package.json 只有 chalk，没有 ansi-styles
import ansiStyles from 'ansi-styles';  // chalk 的依赖，你没声明
# npm 下：能跑（危险但静默）
# pnpm 下：Error: Cannot find module 'ansi-styles'  <- 立刻暴露！

核心差异对比

维度 npm pnpm
存储方式 每个项目独立复制 全局 store + 硬链接
硬盘占用 高（N 个项目 × 包大小） 低（每个版本只存一份）
安装速度 较慢（需复制文件+下载） 快（store 命中则无需下载）
node_modules 结构 铺平（所有层级暴露） 严格（只暴露声明的包）
幽灵依赖 允许存在 直接报错
lock 文件 package-lock.json pnpm-lock.yaml

第四幕：验证与压力测试

反事实：如果 pnpm 的 store 机制不存在

硬盘依然被几十份相同的 lodash 占据；每次在新项目安装已用过的包依然要重新下载；CI/CD 流水线每次构建都要从头下载。

pnpm 解决的不是 npm 解决的问题（依赖管理），而是 npm 解决方式产生的副作用（重复存储、结构宽松）。

边界：pnpm 不是万能的

硬链接有平台限制。 硬链接要求链接源和目标在同一个文件系统分区。如果 store 和项目在不同磁盘，pnpm 会退化为复制文件，节省空间的优势消失。

严格结构有迁移成本。 老项目大量依赖幽灵依赖时，迁移到 pnpm 会"暴露债务"——那些本该在 package.json 里声明的包，得一一补上。

连接：npm → pnpm → 更大的图景

pnpm 的 store 思想，其实和 Docker 镜像的分层缓存、Git 的内容寻址存储是同一个思想：相同的内容，只存一份，用引用代替复制。
